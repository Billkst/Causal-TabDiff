# Causal-tabdiff 项目规则

## 强制执行前置检查
在开始规划或执行任何任务之前，**必须优先检查 `skills` 目录**中是否存在可用 skill。绝对禁止跳过此步骤直接执行任务或编写新代码。

## Conda 环境
本项目的 conda 环境为 `causal_tabdiff`，启动命令为 `conda activate causal_tabdiff`

## 语言强制规则
1. 你必须始终使用中文（简体中文）进行所有回复
2. 即使用户使用英文或其他语言提问，你也必须用中文回答
3. 技术术语、代码变量名、API 名称可保留英文，但必须在首次出现时提供中文解释
4. 禁止在回复中混合使用其他语言进行解释（除非用户明确要求翻译）

## 训练可视化与日志 (强制标准)
为保证长时间实验在前台与后台（如 nohup）场景下均可被实时监控、及时排错并完整追溯，所有训练脚本必须严格遵守以下规范。1. 实时可见性要求所有训练过程必须同时满足终端实时输出与日志文件同步写入。技术要求：终端输出： 必须使用 print(..., flush=True)。文件写入： 必须使用行缓冲模式，例如 open(log_path, "a", buffering=1)。强制刷新： 每次 write 后必须调用 .flush()。启动方式： 建议统一通过 python -u 启动，强制禁用 Python 内部标准流缓冲。2. 进度显示要求交互式环境： 推荐使用 tqdm(..., desc="Training", ncols=80, file=sys.stderr)。非交互环境（nohup/后台）： 建议降低 tqdm 刷新频率或关闭进度条，以防控制字符（^M）污染日志文件。核心原则： 无论是否使用进度条，必须保留纯文本形式的 Epoch 日志。3. 日志输出频率基础状态： 每个 Epoch 至少输出一次。核心指标： 每 10 个 Epoch 至少输出一次。小规模训练： 若总 Epoch $\le 50$，建议每个 Epoch 均输出完整核心指标。4. 日志内容规范4.1 基础信息 (每个 Epoch 必显)每个 Epoch 的日志行必须包含以下字段：Epoch 编号 / 总数随机种子 (Seed)当前学习率 (LR)Train Loss / Val Loss当前最佳指标 (如 Best Val AUPRC)耗时 (Time)推荐格式示例：Epoch 12/100 | Seed 42 | LR 1.0e-3 | TrainLoss 0.1834 | ValLoss 0.2017 | BestValAUPRC 0.0450 | Time 8.2s4.2 核心指标要求必须定期输出以下模型性能评价指标：AUPRC / AUROCF1-Score (若已定义阈值)Precision / Recall5. 后台运行与命名规范正式实验建议使用如下命令启动：Bashnohup python -u run_xxx.py > logs/TRAIN_MODEL_SEED_TIME.log 2>&1 &
日志文件名命名建议：logs/{脚本名}_{模型名}_seed{种子}_{时间戳}.log例子： logs/train_causal_tabdiff_seed42_20260311_1530.log6. 行为准则 (Do's and Don'ts)❌ 严禁行为✅ 推荐做法不带 flush=True 的关键打印使用 print(..., flush=True)训练结束后一次性补写日志实时写入，实时 flush()仅保存最终结果，无过程记录保留每个 Epoch 的状态轨迹仅使用进度条 (tqdm)进度条 + 纯文本 Epoch 日志默认使用 logging 模块 (未配置实时性)优先使用 print 或经配置的 StreamHandler7. 验收标准脚本通过验收需满足：[ ] tail -f 可实时观测日志跳动。[ ] 日志中包含完整的实验配置与 Seed。[ ] 出现异常时，日志能定位到具体的 Epoch 和 Loss 趋势。[ ] 每个阶段的核心指标（AUPRC等）清晰可追溯。
